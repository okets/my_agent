# M6.7-S4 E2E Test Scenarios

> Each dev sprint (S1-S3) adds test scenarios here. S4 implements and runs them all.

---

## From S1: Core Architecture

### Scenario 1: Session resume with fresh system prompt
1. Start dashboard, open web UI
2. Send a message → verify response streams
3. Check server logs: `Starting new SDK session (message 1)`
4. Send a second message → check logs: `Resuming SDK session: <id> (message 2)`
5. Verify system prompt is rebuilt (both messages get fresh prompt)

### Scenario 2: Resume fallback on stale session
1. Send a message (captures SDK session ID)
2. Restart the server (clears in-memory session but DB has stale ID)
3. Send another message → session registry creates new SessionManager with stale SDK ID
4. Resume should fail → fallback to fresh session
5. Check logs: `SDK session resume failed... falling back to fresh session`
6. Verify response still works, new session ID captured

### Scenario 3: /new creates fresh conversation
1. Send a message in current conversation
2. Send `/new`
3. Verify new conversation created, old one in sidebar
4. Send a message → check logs show new SDK session (not resuming old one)

### Scenario 4: System prompt contains expected layers
1. Send a message
2. Inspect server logs or add debug logging to verify prompt contains:
   - Identity content (from brain/CLAUDE.md)
   - Inbound Metadata JSON with channel, conversation_id, message_index
   - Session Context with conversation ID
