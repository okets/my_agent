## FU1 — attachmentId matching between RawMediaStore and channel layer

**Observation:** `rawMediaStore.pathFor(convId, attachmentId)` uses `turn.attachments[].id` as the `attachmentId`. It was not verified during S5 that this `id` matches the `attachmentId` used by the channel layer (`RawMediaStore.write()`) when saving the raw buffer.

**Why not now:** The audio-rescue path degrades gracefully on mismatch: `rawMediaStore.exists()` returns false and the watchdog falls through to text-only rescue rather than crashing. No user-visible breakage; just a missed audio-transcription opportunity.

**Suggested sprint:** S7 (E2E incident replay) — the incident audio test will naturally exercise this path and surface any mismatch.

## FU2 — Voice-placeholder detection string not yet emitted by dashboard

**Observation:** `OrphanWatchdog` detects audio orphans by substring-matching `[Voice note — audio attached, pending transcription]`. This string is generated by `plugins/channel-whatsapp/src/plugin.ts:518` in the WhatsApp plugin. The dashboard's own audio-upload path may not produce this exact string. Detection is forward-compatible (miss → graceful text-only rescue), but the detection coverage is incomplete until the dashboard path is audited.

**Suggested sprint:** S7 or a WhatsApp integration sprint.

## FU4 — reprocessTurn has the same channel-routing bug (carried from S4)

**Observation:** `app.ts:610-627` — `reprocessTurn` (S4's CFR re-drive path) also calls `ci.forwardToChannel(response)` without a `channelOverride`. `CapabilityFailure.triggeringInput.channel.channelId` is available and should be passed in, exactly like the C1 fix applied to the orphan watchdog injector.

**Why not now:** S4 is merged. Out-of-sprint scope per plan §0.4.

**Suggested sprint:** S6 (user-facing messaging). S6's `AckDelivery` work already explicitly uses `TriggeringInput.channel` — the `reprocessTurn` fix is a natural bundle. See architect review FU6.

## FU3 — channel-unification.test.ts harness missing app.cfr mock (carried from S2)

**Observation:** Three tests in `packages/dashboard/tests/integration/channel-unification.test.ts` fail because `AppChatService` is constructed in the test harness without `app.cfr` (the `CapabilityFailureReporter`) being wired. The `chat-service.ts:594` call to `this.app.cfr.emitFailure()` throws `TypeError: Cannot read properties of undefined`. These failures pre-date S5 (exist on master from S2).

**Fix:** Add `app.cfr` as a mock/stub in the channel-unification test harness, or expose a null-safe guard in `chat-service.ts` (`this.app.cfr?.emitFailure(...)`).

**Suggested sprint:** S6 or a dedicated test-debt sprint. Not a blocker for S5 merge.
