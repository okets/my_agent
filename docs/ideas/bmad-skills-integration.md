# BMAD Skills Integration: Deep Dive and Integration Analysis

> **Author:** BMAD Expert (Opus)
> **Date:** 2026-03-04
> **Context:** Comprehensive analysis of the BMAD METHOD ecosystem, its skill/workflow/agent architecture, community modules, and how they relate to my_agent's skills system and the Agent SDK's native skill discovery.

---

## 1. BMAD System Architecture

### 1.1 What BMAD Is

BMAD (Breakthrough Method of Agile AI-driven Development) is an AI-driven development framework (npm: `bmad-method`, v6.0.4) that orchestrates the full software development lifecycle through specialized AI agents, structured workflows, and modular capabilities. It supports Claude Code, Cursor, Codex CLI, and other AI assistants.

**Installation:** `npx bmad-method install` — interactive CLI that creates a `_bmad/` directory in the project root with configuration, agents, workflows, tasks, and data files.

### 1.2 Core Architecture

```
_bmad/                              # Project-local BMAD installation
├── core/                           # Core module (always installed)
│   ├── config.yaml                 # User settings (name, language, output paths)
│   ├── agents/                     # Core agents (orchestrator)
│   ├── workflows/                  # Core workflows (brainstorming, elicitation)
│   └── tasks/                      # Core tasks (reviews, editorial, help)
├── bmm/                            # BMM module (agile development)
│   ├── agents/                     # Role agents (analyst, PM, architect, dev, QA)
│   ├── workflows/                  # Phase workflows (analysis → plan → solution → implement)
│   ├── data/                       # Templates, context files
│   └── templates/                  # Output templates
├── _config/                        # Generated config (agent manifest, etc.)
└── [other-modules]/                # Additional installed modules

.claude/skills/                     # Claude Code skill stubs (generated by installer)
├── bmad-agent-{name}/SKILL.md      # → loads _bmad/{module}/agents/{name}
├── bmad-workflow-{name}/SKILL.md   # → loads _bmad/{module}/workflows/{name}
└── bmad-os-*/SKILL.md              # Open-source utility skills (PR review, triage, etc.)
```

### 1.3 Three Layers

| Layer | Purpose | Example |
|-------|---------|---------|
| **Agents** | Personas with capabilities, communication style, menu of actions | Mary (Analyst), John (PM), Barry (Quick Flow Dev) |
| **Workflows** | Multi-step procedures with micro-file architecture | Create PRD, Brainstorm Project, Quick Spec |
| **Tasks** | Single-purpose operations (reviewable, subagent-delegable) | Adversarial review, editorial review, file audit |

### 1.4 Agent Definitions (YAML)

Each agent is a `.agent.yaml` file with:

```yaml
agent:
  metadata:
    id: "_bmad/bmm/agents/analyst.md"
    name: Mary                    # Agent persona name
    title: Business Analyst
    icon: "..."
    module: bmm
    capabilities: "market research, competitive analysis, ..."

  persona:
    role: Strategic Business Analyst + Requirements Expert
    identity: Senior analyst with deep expertise...
    communication_style: "Speaks with the excitement of a treasure hunter..."
    principles: |
      - Channel expert frameworks: Porter's Five Forces, SWOT...

  menu:                            # Available actions (mapped to workflows)
    - trigger: BP or fuzzy match on brainstorm-project
      exec: "{project-root}/_bmad/core/workflows/brainstorming/workflow.md"
      description: "[BP] Brainstorm Project..."
```

**Key observation:** Each agent has a **fixed persona name** (Mary, John, Barry) and a **fixed communication style**. This is fundamentally incompatible with my_agent's design where the agent's name and personality are set during hatching and must never be overridden.

### 1.5 Workflow Architecture

Workflows use **micro-file architecture**: each step is a separate file with embedded rules, loaded sequentially. The main `workflow.md` serves as an orchestrator that references step files.

Example (brainstorming):
```
workflows/brainstorming/
├── workflow.md              # Orchestrator with initialization and execution flow
├── brain-methods.csv        # 50+ brainstorming techniques (loaded on demand)
├── template.md              # Output template
└── steps/
    ├── step-01-session-setup.md
    ├── step-02-...
    └── ...
```

Workflows are **interactive** — they halt for user input, present menus, and guide users through structured processes. This is conversational, not autonomous.

### 1.6 Task Architecture

Tasks are XML files with structured `<flow>` sections:

```xml
<task id="..." name="Adversarial Review">
  <llm critical="true">
    <i>MANDATORY: Execute ALL steps IN EXACT ORDER</i>
  </llm>
  <flow>
    <step n="1" title="Receive Content">
      <action>Load the content to review</action>
    </step>
    <step n="2" title="Adversarial Analysis">
      <mandate>Review with extreme skepticism</mandate>
    </step>
  </flow>
</task>
```

Tasks are designed to be **delegatable to subagents** — they include all context needed for independent execution.

## 2. BMAD Claude Code Integration

### 2.1 Skill Stubs (`.claude/skills/`)

BMAD's installer generates **thin SKILL.md stubs** in `.claude/skills/` that follow the Agent SDK's native skill format:

```yaml
---
name: bmad-os-review-pr
description: Adversarial PR review tool. Use when user asks to 'review a PR'.
---

Read `prompts/instructions.md` and execute.
```

The stub's entire instruction is a **redirect**: "read the real file and follow it." The actual logic lives in `prompts/instructions.md` (often 100+ lines of detailed procedural instructions).

**This is a clever pattern**: the SDK sees a small, well-described SKILL.md (L1 + minimal L2). When triggered, the skill instructs the agent to read the full instructions from a separate file (effectively L3 resources). Progressive disclosure happens naturally.

### 2.2 Agent Activation via Skills

For agents, the installer generates Claude-specific templates:

```yaml
---
name: 'bmad-analyst'
description: 'Business Analyst — market research, requirements elicitation'
---

<agent-activation CRITICAL="TRUE">
1. LOAD the FULL agent file from {project-root}/_bmad/bmm/agents/analyst.agent.yaml
2. READ its entire contents — complete persona, menu, instructions
3. FOLLOW every step in the <activation> section
4. DISPLAY the welcome/greeting
5. PRESENT the numbered menu
6. WAIT for user input
</agent-activation>
```

**This is where the identity conflict happens.** The activation instruction says "embody this agent's persona" — meaning the AI is told to become "Mary the Business Analyst" with a specific communication style. This directly conflicts with my_agent's hatched personality.

### 2.3 Workflow Skills

```yaml
---
name: 'bmad-brainstorm'
description: 'Brainstorm Project — guided ideation facilitation'
---

LOAD the FULL {project-root}/_bmad/core/workflows/brainstorming/workflow.md,
READ its entire contents and follow its directions exactly!
```

Workflows are more compatible with my_agent since they define procedures, not personas. However, they often reference agent configs for communication style.

## 3. BMAD Module System and Community Skills

### 3.1 Module Architecture

Modules are the primary distribution unit:

```yaml
# module.yaml
code: bmm
name: "BMad Method Agile-AI Driven Development"
description: "AI-driven agile development framework"
default_selected: true

# Module config questions (asked during install)
project_name:
  prompt: "What is your project called?"
  default: "{directory_name}"

user_skill_level:
  prompt: "What is your development experience level?"
  single-select:
    - value: "beginner"
    - value: "intermediate"
    - value: "expert"
```

Modules contain agents, workflows, tasks, data files, and templates. They declare config variables that are resolved at install time.

### 3.2 External Module Registry

BMAD maintains a registry of official external modules in `tools/cli/external-official-modules.yaml`:

| Module | Code | Description | npm Package |
|--------|------|-------------|-------------|
| BMad Builder | bmb | Agent, Workflow and Module Builder | `bmad-builder` |
| Creative Intelligence Suite | cis | Creative tools for writing, brainstorming | `bmad-creative-intelligence-suite` |
| Game Dev Studio | gds | Game development agents and workflows | `bmad-game-dev-studio` |
| Test Architecture Enterprise | tea | Test strategy, automation, release gates | `bmad-method-test-architecture-enterprise` |

### 3.3 Distribution Model

- **npm packages** — Each module is an npm package installable via `npx bmad-method install`
- **GitHub repos** — Source in `bmad-code-org` GitHub organization
- **Module definition** — Each repo has `src/module.yaml` defining config, agents, workflows
- **No central registry** — Modules are registered in `external-official-modules.yaml` (no community marketplace yet)
- **No skill-level distribution** — BMAD distributes **modules** (containing agents + workflows + tasks), not individual skills

### 3.4 Community Contribution Model

BMAD's community model is module-centric, not skill-centric:

1. Fork the BMAD-METHOD repo or create a new module repo
2. Follow the module structure (`src/module.yaml`, agents, workflows)
3. Submit for inclusion in the external modules registry
4. Users install via `npx bmad-method install`

There is **no npm-based skill registry**, no marketplace, and no automatic skill discovery beyond what the BMAD CLI installer generates.

4. **Composable:** Multiple techniques can be combined in a single interaction
5. **License:** MIT — freely adoptable

|-------|---------|------------|
| `bmad-os-review-pr` | Adversarial PR review (Raven's Verdict) | High (multi-step, git checkout, gh CLI) |

| `bmad-os-gh-triage` | Analyze/triage GitHub issues | Medium |
| `bmad-os-findings-triage` | HITL triage of review findings with parallel agents | High |
| `bmad-os-review-prompt` | PromptSentinel v1.2 — audit LLM prompts for failure modes | Very High (3 parallel tracks) |
| `bmad-os-audit-file-refs` | Audit file references using parallel Haiku subagents | Medium |
| `bmad-os-changelog-social` | Generate social media posts from changelog | Medium |

**Format analysis:**

**These are the safest BMAD assets to adopt directly** — they're skill-format-compatible, persona-free, and useful for development workflows.

## 6. Identity and Personality Conflict Analysis

### 6.1 The Core Conflict

BMAD agents have hardcoded personas:

```yaml
persona:
  role: Strategic Business Analyst + Requirements Expert
  identity: Senior analyst with deep expertise...
  communication_style: "Speaks with the excitement of a treasure hunter..."
```

my_agent's design principle: the agent's name and personality are set during hatching (stored in `.my_agent/brain/memory/core/identity.md`). No skill should override this.

### 6.2 Where BMAD Personas Appear

| Component | Has Persona? | Identity Override Risk |
|-----------|-------------|----------------------|
| Agent definitions (`.agent.yaml`) | YES — name, role, communication style | HIGH |
| Agent activation stubs (`.claude/skills/bmad-agent-*`) | YES — "embody this persona" | HIGH |
| Workflows (`.md` files) | Partial — reference `communication_language` from config | LOW |
| Tasks (`.xml` files) | Minimal — "cynical reviewer" is a task-mode persona | LOW |
| Technique CSVs | NONE | NONE |
| OS skills (`bmad-os-*`) | NONE (except Raven's Verdict internal cynical phase) | NONE |

### 6.3 Safe vs Unsafe Adoption

**Safe to adopt directly:**
- Technique libraries (CSV) — no persona at all
- OS skills (`bmad-os-*`) — procedural, no identity override
- Task definitions (XML) — task-mode personas are internal and temporary

**Requires adaptation:**
- Workflows — strip `communication_language` references, use Nina's style
- Agent definitions — extract the **capabilities and menu structure** but discard the persona

**Cannot adopt as-is:**
- Agent activation stubs — fundamentally designed to replace agent identity

### 6.4 Recommended Approach: Extract Capabilities, Discard Personas

BMAD agents are valuable for their **capability descriptions and workflow menus**, not their personas. The adaptation pattern:

```
BMAD Agent                          my_agent Skill
─────────────────                   ──────────────────
persona.name: Mary                  DISCARD
persona.identity: ...               DISCARD
persona.communication_style: ...    DISCARD
persona.principles: ...             EXTRACT → skill instructions
menu[].trigger: BP                  EXTRACT → skill activation
menu[].exec: workflow.md            EXTRACT → workflow reference
menu[].description: ...             EXTRACT → skill description
metadata.capabilities: ...          EXTRACT → skill metadata
```

## 7. Integration Strategies for my_agent

### 7.1 Strategy A: Technique Libraries as Reference Data (Immediate, Low Risk)

**What:** Adopt the brainstorming and elicitation CSVs as reference data that Nina can access when needed.

**How:**
1. Copy `brain-methods.csv` and `methods.csv` to `.my_agent/brain/skills/techniques/`
2. Create a skill `techniques/SKILL.md` that describes the available techniques
3. Nina loads the CSV on-demand when brainstorming or analyzing problems
4. Silent application — Nina uses techniques without announcing them

**Constraints preserved:**
- No persona override — Nina remains Nina
- No identity change — techniques are tools, not characters
- No communication style change — Nina's style from hatching is preserved

**Example interaction:**
```
User: "Help me figure out why our auth system keeps failing"
Nina: (internally selects "5 Whys" + "First Principles Analysis")
Nina: "Let's dig into this. Why is the auth system failing?"
... (guides through root cause analysis without mentioning technique names)
```

### 7.2 Strategy B: Adopt OS Skills (Medium-term, Low Risk)

**What:** Install selected `bmad-os-*` skills directly into my_agent's skill system.

**How:**
1. Copy relevant skills to `.my_agent/.claude/skills/` (if using SDK skills) or `packages/core/skills/` (if using prompt.ts)
2. Adapt the `prompts/instructions.md` to reference my_agent's conventions
3. Remove any BMAD-specific config references

**Best candidates:**
- `bmad-os-review-pr` — directly useful for M7 (Coding Projects)
- `bmad-os-root-cause-analysis` — useful for ops and debugging
- `bmad-os-review-prompt` — useful for skill/workflow development

**Skip:**
- `bmad-os-release-module` — BMAD-specific release process
- `bmad-os-changelog-social` — BMAD-specific social media
- `bmad-os-diataxis-style-fix` — BMAD documentation style

### 7.3 Strategy C: Extract Workflow Procedures (Medium-term, Medium Risk)

**What:** Extract the procedural knowledge from BMAD workflows while discarding persona elements.

**How:**
1. Identify workflows with universal value (not BMAD-specific)
2. Rewrite as Nina-compatible skills (strip persona, keep procedure)
3. Adapt file references to my_agent's directory structure

**Best candidates:**
- **Brainstorming workflow** — universal creative facilitation
- **Quick Spec** — rapid technical specification
- **Code Review** — multi-facet code review
- **Adversarial Review** — cynical review (already persona-free as a task)

**Skip (BMAD-specific):**
- Create PRD — assumes BMAD's artifact structure
- Create Epics and Stories — assumes BMAD's planning model
- Document Project — assumes BMAD's output folder structure

### 7.4 Strategy D: Community Module Integration Framework (Long-term, Medium Risk)

**What:** Build a mechanism for discovering and installing community skills/modules.

**How:**
1. Define a my_agent skill package format (SKILL.md + resources)
2. Create a skill registry (initially curated, later community-contributed)
3. Build a skill installer (validates, audits, installs to `.my_agent/.claude/skills/`)
4. Security: content review, trust tiers, sandboxing hooks

**BMAD learnings to apply:**
- Module-level distribution (not individual files)
- Config questions during install
- Post-install notes
- External module registry pattern

## 8. Comparison: BMAD vs my_agent vs Agent SDK

| Aspect | BMAD | my_agent (current) | Agent SDK Native |
|--------|------|-------------------|-----------------|
| **Skill format** | YAML frontmatter SKILL.md → redirect to instructions | No frontmatter, flat files or subdirs | YAML frontmatter SKILL.md |
| **Discovery** | CLI installer generates stubs | Hardcoded `SKILL_CONTENT_FILES` array | Filesystem scan (`.claude/skills/`) |
| **Loading** | L1 (stub) → L2 (redirect) → L3 (full instructions) | Always-on (full content in system prompt) | L1 (metadata) → L2 (on-demand via Skill tool) |
| **Personas** | Per-agent (hardcoded name, style) | Single hatched identity | No persona concept |
| **Workflows** | XML/Markdown with micro-file steps | Not supported (skills are static text) | Not supported (skills are instructions) |
| **Modules** | npm packages with config questions | None | None |
| **Community** | External module registry (YAML file) | None | Filesystem-based (manual install) |
| **Security** | None (trust the installer) | Guardrails (pre-commit hooks) | None (trust the filesystem) |
| **Token cost** | Low (stubs ~100 tokens, full on-demand) | High (always-on 3-5K per skill) | Low (metadata ~100 tokens, on-demand) |

## 9. Specific Recommendations

### 9.1 Immediate (Pre-Sprint)

1. **Adopt technique CSVs** — Copy brainstorming (50 methods) and elicitation (50 methods) CSVs to `.my_agent/brain/skills/techniques/`. Create a SKILL.md that describes them. Add to `SKILL_CONTENT_FILES` or (better) make them on-demand.

2. **Add YAML frontmatter to existing skills** — Standardize on SDK format now, even before enabling SDK skills. This is 5 minutes of work per skill.

3. **Evaluate `bmad-os-review-pr`** — This is immediately useful. Test it in the project. If it works well, adopt it as a framework skill.

### 9.2 Medium-Term (M7-M8)

4. **Create a `/skill-creator` skill** — A skill that helps users create new skills in the correct format. BMAD's Builder module (`bmad-builder`) does this for BMAD modules. my_agent needs an equivalent for its simpler skill format.

5. **Extract brainstorming workflow** — Adapt BMAD's brainstorming workflow into a Nina-compatible skill. Strip all persona references. Keep the micro-file architecture (workflow references step files). Make technique selection automatic (Nina chooses based on context, not a menu).

6. **Implement selective skill loading in prompt.ts** — Move from always-on injection to metadata-only injection with on-demand loading. This can be done without SDK skills: keep the summaries in system prompt, add a mechanism to load full content when referenced.

### 9.3 Long-Term (M9+)

7. **Define skill package format** — A skill package is a directory with:
   - `SKILL.md` (YAML frontmatter + instructions or redirect)
   - `prompts/` (detailed instructions, if using redirect pattern)
   - `data/` (CSV/JSON reference data)
   - `templates/` (output templates)
   - `manifest.yaml` (dependencies, trust level, required tools)

8. **Build skill installer** — Validates manifest, audits content (no identity overrides, no exfiltration instructions), installs to `.my_agent/.claude/skills/`.

9. **Community registry** — Curated list of tested skills. Start with a YAML file (like BMAD's `external-official-modules.yaml`), evolve to a web-based registry.

## 10. Anti-Patterns to Avoid

### 10.1 Do NOT Import BMAD Agent Personas

BMAD's agent personas ("Mary the Analyst," "John the PM") are deeply embedded in their activation flow. Importing them would cause Nina to "become" someone else. This violates the core design principle.

**Instead:** Extract the capability descriptions and use them as skill metadata.

### 10.2 Do NOT Import BMAD's Menu System

BMAD agents present numbered menus ("1. Brainstorm, 2. Market Research, 3. Domain Research"). This is a BMAD UX pattern that doesn't fit my_agent's conversational style.

**Instead:** Let Nina discover capabilities from skill descriptions and apply them contextually.

### 10.3 Do NOT Import BMAD's Output Folder Structure

BMAD uses `_bmad-output/` with subfolders for each workflow phase. my_agent has its own file structure (`.my_agent/brain/`, `docs/`, etc.).

**Instead:** Map BMAD output patterns to my_agent's existing directory conventions.

### 10.4 Do NOT Import BMAD's Config System

BMAD's `config.yaml` with `user_name`, `communication_language`, etc. duplicates what my_agent handles through hatching. Importing it would create a second source of truth.

**Instead:** Use my_agent's existing identity and preferences system.

### 10.5 Do NOT Announce Technique Usage

BMAD's workflows explicitly present techniques to users ("Choose a brainstorming method: 1. SCAMPER, 2. Six Thinking Hats..."). This is appropriate for BMAD's educational model but not for Nina's "act, don't ask" philosophy.

**Instead:** Nina should silently select and apply techniques based on context. The user should notice better outcomes, not method names.
