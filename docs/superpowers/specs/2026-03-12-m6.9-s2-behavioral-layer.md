## 1. Goal

Make the knowledge lifecycle user-facing. The agent proposes permanent facts for approval, expires unapproved facts, detects stale properties, and runs the morning brief with higher-judgment models. Morning brief timing becomes a user preference in `config.yaml`.

**What S1 built:** Classification pipeline, staging CRUD, properties utilities, `queryModel()`, summary rollup chain, morning prep job.

**What S2 adds:** The behavioral layer on top — approval flow, staleness detection, model upgrade, user-configurable preferences.

## 2. Config Schema Extension

### 2.1 `config.yaml` — New `preferences` Section

```yaml
preferences:
  morningBrief:
    time: "08:00"           # HH:MM in user's timezone
    model: sonnet           # haiku | sonnet | opus
    channel: default        # S3: which channel to initiate on (unused until S3)
  timezone: Asia/Bangkok    # user's timezone, not dynamic status
```

**Defaults:** `time: "08:00"`, `model: "sonnet"`, `channel: "default"`, `timezone: "UTC"`.

**Timezone note:** `preferences.timezone` is the static fallback (home timezone). The scheduler prefers the dynamic `properties/status.yaml` timezone when available — so if the user is traveling and extraction updates the timezone property, the morning brief fires at 08:00 local time wherever they are. See Section 2.3.

### 2.2 Config Loading

Extend `loadConfig()` in `packages/core/src/config.ts` to parse the `preferences` section. New export:

```typescript
interface MorningBriefPreferences {
  time: string;           // "HH:MM"
  model: string;          // "haiku" | "sonnet" | "opus" (plain string, not ModelAlias — core can't import from dashboard)
  channel: string;        // "default" or channel ID
}

interface UserPreferences {
  morningBrief: MorningBriefPreferences;
  timezone: string;       // IANA timezone
}

function loadPreferences(agentDir: string): UserPreferences
```

Returns defaults for any missing fields. Never throws.

**Note:** `model` is typed as `string` in core. The dashboard's `queryModel()` casts to `ModelAlias` at the call site. This avoids a core→dashboard dependency.

### 2.3 Scheduler Migration

The morning-prep entry in `work-patterns.md` is removed. The scheduler reads morning brief time from `loadPreferences().morningBrief.time` instead. All other jobs (daily-summary, weekly-summary, monthly-summary) stay in `work-patterns.md`.

**Timezone-aware scheduling:** The existing `isDue(cadence, lastRun, now)` in `work-patterns.ts` uses `new Date()` system-local time. Morning prep bypasses `isDue()` entirely — the scheduler computes the target time in the user's **current** timezone:

**Timezone resolution order:**
1. `properties/status.yaml` `timezone` value (dynamic — updated by extraction or agent during conversation)
2. `config.yaml` `preferences.timezone` (static fallback — set during hatching)

This means when the user travels and the extraction pipeline updates `status.yaml` with a new timezone, the morning brief automatically shifts to their local time. No manual config change needed.

```typescript
// In WorkLoopScheduler, dedicated morning-prep check:
function isMorningPrepDue(
  preferences: UserPreferences,
  properties: PropertiesMap,
  lastRun: Date | null,
  now: Date,
): boolean {
  // Prefer dynamic timezone from properties, fall back to static preference
  const timezone = properties.timezone?.value ?? preferences.timezone;

  const formatter = new Intl.DateTimeFormat("en-US", {
    timeZone: timezone,
    hour: "2-digit", minute: "2-digit", hour12: false,
  });
  const localTime = formatter.format(now);  // "08:00" etc.
  const targetTime = preferences.morningBrief.time;

  // Due if: local time >= target AND (never ran OR last ran before today's target in user's TZ)
  return localTime >= targetTime && (!lastRun || lastRun < todayTargetInUserTZ(timezone, now));
}
```

No changes to `isDue()` — it stays untouched for work-pattern jobs. Uses Node.js built-in `Intl` (no new dependencies).

## 4. Staged Fact Proposal in Morning Brief

### 4.1 Current State

Morning prep already reads staging files and includes them in context assembly (work-loop-scheduler.ts lines 533-543). But the current prompt doesn't instruct the model to format them as proposals.

### 4.2 Change

Update the morning prep user prompt template to include a dedicated section for staged facts:

```
{staged_facts_section}
```

When staging files exist, this section renders as:

```
PENDING KNOWLEDGE — propose these to the user naturally:
- [user-info] Has two daughters, Noa (5) and Maya (3) (source: conv "Thailand Planning", attempts: 0)
- [contact] Kai — tour guide in Chiang Mai (source: conv "Thailand Planning", attempts: 1)

For each pending fact, include a brief natural proposal in today's section.
Example: "I noted you mentioned Noa and Maya — shall I add them to your profile?"
If no pending facts, omit this section entirely.
```

When no staging files exist, `{staged_facts_section}` renders as empty string.

### 4.3 Output Format

The morning brief output (`current-state.md`) gains a "Pending" subsection under "Today" when staged facts exist:

```markdown
## Today — March 13

### Pending
- You mentioned Noa (5) and Maya (3) — shall I add them to your profile?
- Kai was your tour guide in Chiang Mai — worth saving to contacts?

### Events
- No meetings
- S2 sprint in progress
```

The agent sees this in its system prompt and proposes the facts conversationally.

## 5. `manage_staged_knowledge` MCP Tool

### 5.1 Purpose

Safe mechanical tool for the agent to approve, reject, or skip staged facts. The agent calls this when the user responds to a proposal.

### 5.2 Tool Definition

```typescript
tool(
  "manage_staged_knowledge",
  "Approve, reject, or skip a staged knowledge fact. " +
  "Approve writes the fact to the appropriate reference file. " +
  "Reject deletes it from staging. " +
  "Skip increments the attempts counter.",
  {
    action: { type: "string", enum: ["approve", "reject", "skip"] },
    stagingFile: { type: "string", description: "Path to the staging file" },
    factText: { type: "string", description: "The exact text of the fact to act on (matched against staging file content)" },
    enrichment: { type: "string", description: "Additional detail from user to merge into the fact (optional)" },
  },
  handler
)
```

**Fact identification by text match, not index.** Index-based addressing is fragile — if a fact is deleted (approved/rejected), subsequent indices shift. Text matching against the staging file content is stable across concurrent operations. The match is substring-based on the fact text (ignoring the `[subcategory, attempts: N]` prefix).
```

### 5.3 Action Behavior

**approve:**
1. Find the staged fact matching `factText` in `stagingFile`
2. Determine destination from `subcategory`:
   - `user-info` → `notebook/reference/user-info.md`
   - `contact` → `notebook/reference/contacts.md`
   - `preference:personal` → `notebook/reference/preferences/personal.md`
   - `preference:work` → `notebook/reference/preferences/work.md`
   - `preference:communication` → `notebook/reference/preferences/communication.md`
3. If `enrichment` is provided, append it to the fact text
4. Append the fact to the destination file (create if doesn't exist, append under appropriate heading)
5. Remove the fact from the staging file
6. If staging file has no remaining facts, delete it
7. Return `{ approved: true, destination: "<path>" }`

**reject:**
1. Find and remove the fact matching `factText` from `stagingFile`
2. If staging file has no remaining facts, delete it
3. Return `{ rejected: true }`

**skip:**
1. Find the fact matching `factText` and increment its attempts counter
2. Return `{ skipped: true, attempts: <new count> }`

### 5.4 Error Handling

- Invalid `stagingFile` path → error with message
- `factIndex` out of range → error with message
- Destination file write failure → error, staging file unchanged (no partial state)

### 5.5 Registration

The tool lives in `packages/dashboard/src/mcp/tools/manage-staged-knowledge.ts` — NOT in core. It imports `knowledge-staging.ts` functions which are dashboard-local. Register on the brain's MCP server via the existing tool registration pattern in the dashboard's MCP setup. Available to Conversation agent only (not working agents).

## 6. Approval Retry and Expiry

### 6.1 Retry Logic

Each morning brief run processes all staging files:
1. Read all staging files via `readStagingFiles()`
2. For each file, for each fact:
   - If `attempts >= 3` → delete the fact (expired, user ignored it)
   - Otherwise → include in the morning brief prompt
3. After morning brief completes, increment attempts for all facts that were proposed but not yet acted on

### 6.2 Expiry Cleanup

Run as part of `handleMorningPrep()`, before the morning brief query:
1. Read all staging files
2. Delete facts with `attempts >= 3`
3. Delete empty staging files
4. Log expired facts count

### 6.3 Attempt Increment: Morning Brief Auto-Increments, Not the Agent

**Morning brief auto-increments all proposed facts** after the brief runs. This is the primary mechanism — it tracks "how many times was this fact presented." The MCP tool's `skip` action is for explicit "not now" responses from the user (also increments, so user-skipped facts expire faster).

This means a fact proposed in morning brief and then explicitly skipped by the user gets +2 attempts that cycle (auto +1, skip +1). This is intentional — explicit rejection is stronger signal than silence.

### 6.4 Staging File Operations

The current `incrementAttempts()` function increments ALL facts in a file. S2 adds per-fact operations. Update `knowledge-staging.ts`:

```typescript
// Find a fact by text match (substring of fact text, ignoring metadata prefix)
function findStagedFact(facts: StagedFact[], factText: string): number  // returns index or -1

// Increment attempts for a specific fact (by text match)
async function incrementFactAttempts(
  filePath: string,
  factText: string,
): Promise<number>  // returns new attempts count

// Increment attempts for ALL facts in a file (used by morning brief auto-increment)
async function incrementAllAttempts(
  filePath: string,
): Promise<void>

// Delete a specific fact from a staging file (by text match)
async function deleteStagedFact(
  filePath: string,
  factText: string,
): Promise<void>  // deletes file if no facts remain

// Delete all expired facts (attempts >= maxAttempts)
async function cleanExpiredFacts(
  agentDir: string,
  maxAttempts: number,
): Promise<number>  // returns count of deleted facts
```

## 7. Property Staleness Detection

### 7.1 Thresholds

From design spec §6.5:

| Property | Staleness Threshold | Rationale |
|----------|-------------------|-----------|
| `location` | 7 days | Trips end, people move |
| `timezone` | 30 days | Changes with location, less frequently |
| `availability` | 3 days | Status changes rapidly |

Thresholds are hardcoded constants (not user-configurable). New properties default to 30 days.

### 7.2 Detection

New function in `packages/dashboard/src/conversations/properties.ts`:

```typescript
interface StaleProperty {
  key: string;
  value: string;
  confidence: string;
  daysSinceUpdate: number;
  threshold: number;
}

function detectStaleProperties(
  properties: PropertiesMap,
  today: string,  // ISO date
): StaleProperty[]
```

### 7.3 Morning Brief Integration

Stale properties are included in the morning brief prompt alongside staged facts:

```
STALE PROPERTIES — ask the user if these are still current:
- location: "Chiang Mai, Thailand" (last updated 10 days ago, threshold: 7 days)
- availability: "vacation" (last updated 5 days ago, threshold: 3 days)
```

The morning brief output includes these as questions in the "Today" section:

```markdown
### Check-in
- Still in Chiang Mai? (last confirmed March 5)
- Are you still on vacation?
```

The agent uses `notebook_write` to update properties when the user confirms or corrects. The `updateProperty()` utility in `properties.ts` is not exposed as an MCP tool in S2 — it's an internal function used by the extraction pipeline. Exposing it as a tool is deferred to the M6.8 knowledge curation skill.

## 8. Hatching and Settings Integration

### 8.1 Hatching

Add morning brief preferences to the hatching flow. This can be a sub-question in the existing operating rules step or a new lightweight step:

- "What time would you like your morning brief?" (default: 08:00)
- "What's your timezone?" (must be asked — agent runs on VPS, not user's machine, so system timezone is irrelevant)

Write to `config.yaml` under `preferences:`.

Model selection (`sonnet` default) is not asked during hatching — it's an advanced setting.

### 8.2 Settings API

New endpoints or extend existing settings routes:

```
GET  /api/settings/preferences → UserPreferences
PUT  /api/settings/preferences → update config.yaml preferences section
```

### 8.3 Settings UI

Add a "Morning Brief" section to the settings page:
- Time picker for morning brief time
- Timezone selector
- Model selector (haiku/sonnet/opus)

Design follows existing settings UI patterns (glass-strong panels, Tokyo Night palette).
