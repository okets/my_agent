## 2. S3 Gap Fixes

### 2.1 alert() Channel Routing

**Problem:** `alert()` calls `trySendViaChannel()` which uses the global `getOutboundChannel()` preference. The spec says: if the active conversation is on web, send via web — respect the active conversation's channel.

**Fix:** `alert()` already has the active conversation object. Pass the active conversation's channel to `trySendViaChannel()` instead of always using the global preference. `trySendViaChannel()` gains an optional `channelOverride` parameter.

**Files:** `conversation-initiator.ts`

### 2.2 firstTurnPrompt Wiring

**Problem:** S3 reviewer flagged that `initiate()` accepts `firstTurnPrompt` but it wasn't passed through to `streamNewConversation()`.

**Fix:** Verify current state — the code at `conversation-initiator.ts:138-141` does pass `prompt` to `streamNewConversation()`. If the wiring is already correct, close this as resolved. If not, wire it through.

**Files:** `conversation-initiator.ts`, verify `index.ts` factory

## 3. Task Completion Notifications

### 3.1 New Field: `notifyOnCompletion`

Add to `Task` interface and `CreateTaskInput`:

```typescript
notifyOnCompletion?: "immediate" | "debrief" | "none"
```

**Defaults (applied at creation if not specified):**
- `immediate` tasks → `"immediate"`
- `scheduled` tasks → `"debrief"`

The brain can override per-task when creating via MCP tool.

**DB migration:** Add `notify_on_completion TEXT DEFAULT NULL` column to tasks table. NULL = use type-based default.

### 3.2 TaskProcessor Integration

`TaskProcessor.deliverResult()` currently:
1. Appends result to conversation transcript
2. Broadcasts via WebSocket
3. Fires notification service

**New behavior based on `notifyOnCompletion`:**

- **`"immediate"`**: After appending to transcript + broadcasting, call `CI.alert()` with a structured prompt: task title + result summary. If no active conversation, call `CI.initiate()`. The brain interprets and presents the result in Conversation Nina's voice.
- **`"debrief"`**: Append to transcript + broadcast only (no CI outreach). Task appears in next debrief.
- **`"none"`**: Append to transcript + broadcast only. No debrief inclusion either.

**Resolved `notifyOnCompletion`:** Since the field is nullable (NULL = use default), `TaskProcessor` resolves the effective value: `task.notifyOnCompletion ?? (task.type === "immediate" ? "immediate" : "debrief")`.

### 3.3 ConversationInitiator Dependency

`TaskProcessor` gains `ConversationInitiator` as an optional dependency (injected via config). When null, falls back to current behavior (no CI outreach).

**Files:** `task-processor.ts`, `core/src/tasks/types.ts`, task DB migration, `index.ts` (wiring)

## 5. On-Demand Debrief (MCP Tool)

### 5.1 Tool Definition

New MCP tool `request_debrief` on the brain's MCP server:

```typescript
tool("request_debrief", "Run or retrieve today's debrief", {}, handler)
```

No parameters.

### 5.2 Behavior

1. Check if `debrief-prep` has a successful run today in the `work_loop_runs` table (consistent with existing duplicate guard pattern)
2. If **yes** → read `notebook/operations/current-state.md` and return content
3. If **no** → trigger `handleDebriefPrep()` synchronously, return the output

The tool returns structured data (the debrief markdown). The brain (Conversation Nina) interprets and presents it conversationally.

### 5.3 Integration

The tool is registered on the existing brain MCP server alongside `remember`, `recall`, etc. The `WorkLoopScheduler` (which owns `handleDebriefPrep()`) is passed to the MCP server factory.

**Files:** New tool in `packages/dashboard/src/mcp/` (co-located with existing knowledge MCP tools, depends on `WorkLoopScheduler`), `work-loop-scheduler.ts` (expose method)

## 6. Long-Running Response Notifications

### 6.1 WhatsApp Typing Indicator Refresh

**Current:** Single `sendTypingIndicator()` call before processing starts.

**Change:** Start an interval that calls `sendTypingIndicator()` every 10 seconds. Clear on first `text_delta` received or on response completion/error.

**Where:** `message-handler.ts`, in the owner message handling flow after the initial `sendTypingIndicator()` call.

### 6.2 Interim Messages

Two timed interim messages when the first token hasn't arrived:

| Timer | Delay | Message |
|-------|-------|---------|
| 1st | 30s | Short reassurance (e.g., "Working on it...") |
| 2nd | 90s | Second nudge (e.g., "Still on it, bear with me...") |

**Max 2 interim messages.** After that, only typing indicator dots continue.

**All timers clear** on first `text_delta` event, response completion, or error.

### 6.3 Channel Behavior

**WhatsApp:**
- Interim messages sent as real WhatsApp messages via `channelManager.send()`
- Typing indicator refreshes every 10s throughout
- Interim messages are NOT appended to the conversation transcript — ephemeral, for UX only

**Web dashboard:**
- New WebSocket message type: `{ type: "interim_status", message: string }`
- UI shows a subtle system-style message that disappears when real tokens start streaming
- Existing `start` event already triggers typing dots in the UI

### 6.4 Implementation Location

New utility: `ResponseTimer` class (or similar) that encapsulates:
- Typing indicator refresh interval (10s)
- Interim message timeouts (30s, 90s)
- Cleanup on cancel

Instantiated in `message-handler.ts` per incoming message. Receives `sendTypingIndicator` and `sendInterimMessage` callbacks. Caller invokes `timer.cancel()` when streaming starts or completes.

**Files:** New `response-timer.ts` utility, `message-handler.ts`, `ws/protocol.ts` (new message type), `public/js/app.js` (handle `interim_status`), `public/index.html` (render interim message)

## 8. Edge Cases

| Scenario | Behavior |
|----------|----------|
| Task completes while user is in active conversation | `CI.alert()` injects result into active conversation |
| Task completes with no active conversation, `notifyOnCompletion = "immediate"` | `CI.initiate()` starts new conversation on preferred channel |
| Task completes with `notifyOnCompletion = "debrief"`, no debrief scheduled | Result waits until user requests debrief via MCP tool or next scheduled run |
| `request_debrief` called twice in same day | Second call reads cached `current-state.md` (no re-run) |
| Interim message sent but response arrives 1s later | Interim message already sent (ephemeral), real response follows normally |
| WhatsApp disconnected during interim message | `trySendViaChannel` catch block silently fails, web still works |
| Multiple tasks complete simultaneously with `"immediate"` | Each triggers `CI.alert()` sequentially — brain receives multiple system turns |
