# M9.2: Worker Todo Coverage — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extend M9.1's code-enforced Todo system to all worker job types — not just capabilities. Every Working Nina gets a baseline checklist with validators, including a visual chart gate for data-rich output. Also: make the visual augmentation hook smarter so it only fires when the brain genuinely forgot, not when it deliberately decided against charting.

**Origin:** Filtered from the [Cognitive Todo Boost proposal](../superpowers/plans/2026-04-06-cognitive-todo-boost-proposal.md). See the Verdict section there for why 6 of 10 tasks were dropped.

**Principle:** Working Ninas are autonomous — no user in the loop. Code enforcement (validators, completion gating, mandatory items) is the only way to ensure compliance. This is M9.1's proven finding, applied to the gaps M9.1 didn't cover.

**Validation method:** M9.1-S8 pattern — real LLM smoke tests after each sprint. No mocks. Unit tests for code, real Agent SDK sessions for behavior.

**Design spec:** [agentic-flow-overhaul.md](../design/agentic-flow-overhaul.md) (M9.1 spec — M9.2 extends it, no new design doc needed)

### Review Fixes Applied

This plan was independently reviewed. Three blocking issues and one feature addition were identified and fixed:

1. **`"research"` added to type system** — S1 now includes updating `AutomationManifest.job_type` union and the Zod enum in `automation-server.ts`, without which the research template would be unreachable via the MCP tool.
2. **Smoke tests use disk-write pattern** — all smoke tests now follow M9.1's proven pattern (write YAML manifest to disk → restart dashboard → fire via `/api/automations/:id/fire`), not the non-existent `POST /api/automations` endpoint.
3. **Chart todo item guards `visualService`** — the research template chart item text now accounts for workers that may not have `create_chart` available.
4. **Smart visual hook** — S4 expanded to include a Haiku pre-check gate so the fallback only fires when the brain forgot, not when it deliberately skipped charting (prevents dumb charts of unrelated data).

# Visual Expression

You have two tools for visual communication. Use them **proactively** — don't wait to be asked. A text-only response for data-rich content is an incomplete response.

## Decision Tree (check on every response)

` ` `
Does my response contain data?
├── 3+ numeric data points (counts, scores, prices, measurements) → MUST call create_chart
├── Comparison across categories or time periods → MUST call create_chart
├── Status with numeric values (progress %, ratings) → SHOULD call create_chart
└── No numeric data
    ├── Topic has a visual component (place, product, weather) → call fetch_image
    └── Purely conversational → no visual needed
` ` `

**Why this matters:** If you skip `create_chart`, a Haiku fallback generates a worse chart in a separate message. Your chart is better (Opus quality, inline in your response). Own it.

## create_chart Protocol

1. Generate SVG following the rules below
2. Call `create_chart` with the SVG and a description
3. Embed the returned URL as `![description](url)` in your response text
4. If you don't embed `![...](url)`, the user sees nothing

### SVG Rules
- Dimensions: `<svg xmlns="http://www.w3.org/2000/svg" width="600" height="350">`
- Inline `style=""` attributes only — NO `<style>` blocks
- Font: `sans-serif` only — no custom or web fonts
- No `<foreignObject>` elements
- Round corners on background: `rx="12"`

### Color Palette (Tokyo Night)
| Token | Hex | Use |
|-------|-----|-----|
| background | `#1a1b26` | Chart background |
| panel | `#292e42` | Data area, legend bg |
| text | `#c0caf5` | Labels, values |
| muted | `#565f89` | Grid lines, axes |
| accent | `#7aa2f7` | Primary data series |
| purple | `#bb9af7` | Secondary series |
| pink | `#f7768e` | Negative/alert values |
| green | `#9ece6a` | Positive/success values |
| yellow | `#e0af68` | Warning/third series |

## fetch_image Protocol

1. Call `fetch_image` with the image URL and description
2. Embed the returned URL as `![description](url)` in your response text
3. Use for: weather maps, product photos, location images, news photos
4. Do NOT use for: generic stock photos, decorative images

## Constraints
- Max 3 images per response
- Images augment text — always include a text explanation alongside
- If unsure whether data warrants a chart, err toward generating one
